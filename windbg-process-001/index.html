Windows のプロセス

Windows におけるプロセスは、基本的に 1 つ以上のスレッドを持つコンテナとして作成されます。

OS の中では、プロセスは エクゼクティブプロセス(EPROCESS) 構造体によって表現されています。

この中には、プロセスに関する属性や、関連するデータ構造体へのポインタが含まれており、EPROCESS 構造体がプロセスオブジェクトとして機能します。(エクゼクティブのオブジェクトマネージャが EPROCESS 構造体を 1 つのプロセスオブジェクトとしてカプセル化しています)

EPROCESS 構造体の情報自体はカーネルデバッガでdt nt!_EPROCESSコマンドを実行すると参照できます。

出力結果については一応こちらに添付してますが、概要レベルで把握するだけならインサイド Windows の図 3-2 を参照するのがよいかと思います。

EPROCESS の構造

EPROCESS 構造体は、最初のフィールドとしてプロセス制御ブロック(PCB)を持ちます。

PCB とは、カーネルプロセスのために使用される KPROCESS 型の構造体です。

KPROCESS には、システムカーネルに存在するディスパッチャー、スケジューラ、割込みなどの情報が格納されます。

EPROCESS 構造体の保持するデータの大部分はシステムのアドレス範囲に存在しています。

そのため、ユーザモードのプロセスダンプなどでは.processなど EPROCESS 構造体の情報を参照するコマンドを使用できません。

ただし、ユーザモードからアクセスされる各プロセスの PEB のみはユーザモードのアドレス範囲に格納されます。

EPROCESS 構造体からシステム内でのプロセス情報を列挙する

前述の通り、EPROCESS 構造体のActiveProcessLinksは、OS 内で稼働するすべてのプロセスを連結する双方向リストです。

そのため、ActiveProcessLinksの情報を列挙することで、システム内のすべてのプロセス情報を参照することもできます。(一部のマルウェアなどはActiveProcessLinksのリストを改ざんすることで、不正なプロセスを隠そうとする場合があります)

ActiveProcessLinksのリストをダンプするため、まずはnt!PsActiveProcessHeadの情報を取得します。

nt!PsActiveProcessHeadには、システムで一番初めに作成されるシステムプロセスの ActiveProcessLinksへのポインタが格納されます。

つまり、nt!PsActiveProcessHead内のポインタが示すアドレスから、ActiveProcessLinksのオフセットを引くことで、システムプロセスと対応する EPROCESS 構造体のアドレスを取得できます。

dt コマンドの -l オプションにリストのアドレスを与えることでそのリストをたどって情報をダンプすることができるので、以下のコマンドを順に実行することでシステム内の EPROCESS 構造体を列挙して.process 0 0と同等の出力を得ることができるようになります。

# System プロセスの ActiveProcessLinks へのポインタアドレスを取得
r $t0 = nt!PsActiveProcessHead;r $t1 = poi(@$t0);r $t1

# ActiveProcessLinks.Flink のオフセットを参照
dt nt!_EPROCESS ActiveProcessLinks
> +0x448 ActiveProcessLinks 

# _EPROCESS をダンプ(.process 0 0 と同等の出力)
dt nt!_EPROCESS -l ActiveProcessLinks.Flink -y Ima -yoi Uni $t1-0x448

参考：dt (Display Type) - Windows drivers | Microsoft Learn

また、プロセス情報を列挙する のコマンドを使用することで、システム内のコマンドを探索することもできます。

以下は参考例です。

# システム内のすべてのプロセスを列挙
!process 0 0

# 特定のプロセスの完全な詳細情報を取得
!process 0 7 notepad.exe

# デバッガのコンテキストを指定のプロセスに変更する(！peb などが使えるようになる)
.process /r /P <!process で特定した EPROCESS ブロックのアドレス>

# すべてのプロセスを探索する(すべてのプロセスに対してコマンドを発行することも可能)
!for_each_process

保護されたプロセスを特定する

Windows では、特定のプロセスを保護されたプロセスとして扱います。

プロセスが保護されている場合、Debug privilegeを持つ場合でも、保護されたプロセスのメモリの読み書きやスレッドの停止などのデバッグ操作が大きく制限されます。

保護されたプロセスの仕組みは、システムプロセスやセキュリティソフトウェア、また DRM などで保護されるメディアなどで使用されます。

参考：Debug Privilege - Windows drivers | Microsoft Learn

プロセスがどのような保護レベルにあるかは、Process Explorer から以下のように特定できます。

CreateProcess API を呼び出す

インサイド Windows によると、ドキュメント化されているすべてのプロセス作成関数は、Kernel32.dll で定義されているCreateProcessInternalWというネイティブ API を呼び出すそうです。

プロセス作成関数には、プロセス自身と同じアクセストークンを使用してプロセスを起動すCreateProcess関数や、指定したアクセストークンを使用してプロセスを起動するCreateProcessAsUser関数などがあります。(他にも、CreateProcessWithTokenWやCreateProcessWithLogonWなどもあります)

Docs の記載を参考に、CreateProcess関数を呼び出すだけのコードを作成しました。

#include <stdio.h>
#include <windows.h>

void main()
{
    STARTUPINFO si;
    PROCESS_INFORMATION pi;

    WCHAR t1[] = L"c:\\Windows\\System32\\notepad.exe";
    LPWSTR lpCommandLine = t1;

    ZeroMemory(&si, sizeof(si));
    si.cb = sizeof(si);
    ZeroMemory(&pi, sizeof(pi));

    /*
    BOOL CreateProcessW(
      [in, optional]      LPCWSTR               lpApplicationName,
      [in, out, optional] LPWSTR                lpCommandLine,
      [in, optional]      LPSECURITY_ATTRIBUTES lpProcessAttributes,
      [in, optional]      LPSECURITY_ATTRIBUTES lpThreadAttributes,
      [in]                BOOL                  bInheritHandles,
      [in]                DWORD                 dwCreationFlags,
      [in, optional]      LPVOID                lpEnvironment,
      [in, optional]      LPCWSTR               lpCurrentDirectory,
      [in]                LPSTARTUPINFOW        lpStartupInfo,
      [out]               LPPROCESS_INFORMATION lpProcessInformation
    );
    */
    
    if (!CreateProcess(
        NULL,           // No module name (use command line)
        lpCommandLine,  // Command line
        NULL,           // Process handle not inheritable
        NULL,           // Thread handle not inheritable
        FALSE,          // Set handle inheritance to FALSE
        0,              // No creation flags
        NULL,           // Use parent's environment block
        NULL,           // Use parent's starting directory 
        &si,            // Pointer to STARTUPINFO structure
        &pi)            // Pointer to PROCESS_INFORMATION structure
        )
    {
        printf("CreateProcess failed (%d).\n", GetLastError());
        return;
    }

    // Wait until child process exits.
    WaitForSingleObject(pi.hProcess, INFINITE);

    // Close process and thread handles. 
    CloseHandle(pi.hProcess);
    CloseHandle(pi.hThread);
}

OpenProcess API で既存のローカル Process オブジェクトを開く

OpenProcess API を使用することで既存のローカルプロセスオブジェクトのハンドルを取得できます。

HANDLE OpenProcess(
  [in] DWORD dwDesiredAccess,
  [in] BOOL  bInheritHandle,
  [in] DWORD dwProcessId
);

例えば、以下の例では PID を指定してプロセスオブジェクトのハンドルを取得します。

以下の例では要求するアクセス権限としてはPROCESS_ALL_ACCESSを指定しています。

このアクセス権は通常、プロセスのセキュリティ記述子と検証されますが、呼び出しユーザにSeDebugPrivilegeの特権が存在する場合は必ず要求されたアクセスが許可されます。

#include "windows.h"
#include "tchar.h"
#pragma comment(lib, "advapi32.lib")

int main()
{

	//HANDLE OpenProcess(
	//	[in] DWORD dwDesiredAccess,
	//	[in] BOOL  bInheritHandle,
	//	[in] DWORD dwProcessId
	//);

	// 疑似プロセスオブジェクトの取得
	HANDLE hProcess = NULL;
	hProcess = GetCurrentProcess();

	// Try to open the remote process.
	DWORD PID = <PID>;
	hProcess = OpenProcess(
		PROCESS_ALL_ACCESS,
		TRUE,
		PID);

	if (!hProcess)
	{
		wprintf(L"ERROR: Could not get a handle\n");
		return 1;
	}
	wprintf(L"[+] Got process handle.\n");


	// Create a pointer to a Token
	PHANDLE pToken = new HANDLE;

	BOOL bResult = OpenProcessToken(
		hProcess, 
		TOKEN_QUERY | TOKEN_IMPERSONATE, 
		pToken);

	if (!bResult)
	{
		wprintf(L"ERROR: Could not open process token.\n");
		return 1;
	}
	wprintf(L"[+] Process token is now open.\n");

    
	// GetProcessId
	DWORD cPID = GetProcessId(hProcess);
	wprintf(L"[+] GetProcessId result: %d.\n", cPID);

	return 0;
}
